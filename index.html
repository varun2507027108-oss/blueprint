1. System Architecture & Tech Stack Rationale

Blueprint is structured as a decoupled web application with separated responsibilities:

Frontend Next.js 15 (App Router), Tailwind CSS, Framer Motion, React Flow

Next.js handles Server-Side Rendering (SSR) and client routes out of the box. Framer Motion provides the premium animations (auto-advancing tabs, hover reveals). React Flow is used to render an interactive, pan-and-zoomable database ERD diagram from raw AI text by parsing the Mermaid schema.

Backend FastAPI, Pydantic V2, Uvicorn

FastAPI is asynchronous by default, allowing it to handle hundreds of concurrent status-polling requests without blocking the event loop. Pydantic V2 enforces strict type checks, ensuring the AI outputs always align with the expected JSON structure.

Orchestration LangGraph

Instead of simple linear prompt chains, LangGraph models the workflow as a state machine. This enables parallel branching (Architect and Marketing run simultaneously) and Human-in-the-Loop (HITL) interrupts (pausing execution for user approval).

LLM Hardware Groq Llama 3.3 70B (LPU Inference)

Groq leverages specialized LPU (Language Processing Unit) hardware, yielding near-zero latency. It executes the entire 6-agent pipeline in under 30 seconds for free, whereas standard providers would take minutes and incur high transaction costs.

2. Deep Dive: LangGraph & State Mechanics

LangGraph coordinates state using a centralized state dictionary (GraphState). Unlike standard sequential wrappers, it uses strict merge operators and checkpointers to manage threads.

A. GraphState Reducers

When the graph branches into parallel paths (Growth Stream and Engineering Stream), multiple nodes try to update the state at the same time. We use Pydantic annotations with reducer functions: 

def reduce_dict(left: Optional[Dict[str, Any]], right: Optional[Dict[str, Any]]) -> Dict[str, Any]:
    merged = dict(left or {})
    merged.update(right or {})
    return merged

class GraphState(BaseModel):
    stages: Annotated[Dict[str, Any], reduce_dict] = Field(default_factory=dict)

This tells LangGraph that instead of overwriting the stages variable when parallel threads update it, it must pass the old dict (left) and the new dict (right) to reduce_dict to merge them safely.

B. The Human-in-the-Loop Interrupt Flow

If the Startup Advisor flags a risky idea (risk_score > 0.85), it calls LangGraph's interrupt() function.

  1. The call raises a GraphInterrupt exception.
  2. LangGraph halts the thread and writes the current variables and program counter to checkpoints.db.
  3. The FastAPI thread terminates, returning awaiting_gate status.
  4. The user clicks "Force Approve". The frontend calls POST /sessions/{id}/gate-decision.
  5. The backend loads the state from SQLite, issues Command(resume={"decision": "continue"}), and LangGraph resumes execution from the exact line where it paused.

3. The 6 Agents & Output Contracts

Each agent is guided by a McKinsey/VC-caliber system prompt, and outputs are strictly validated by Pydantic contracts.

1. Startup Advisor Llama 3.3 70B @ Groq

Lenses: Founder-Market Fit, Market Timing, Unit Economics, Competitive Moat, Single Point of Death. Outputs: ValidationResult (verdict: str, risk_score: float, reasoning: str, red_flags: list).

2. Market Researcher Llama 3.3 70B @ Groq + Tavily API

Queries Tavily for competitor details and McKinsey TAM/SAM/SOM statistics. Outputs: MarketResearchReport (tam_estimate: str, competitors: list[Competitor], trends: list[str], SWOT, gaps).

3. Product Manager Gemini 2.5 Flash @ Google

Synthesizes concept logs. Uses MoSCoW prioritization and writes Jobs-to-be-Done (JTBD) user stories. Outputs: PRD (problem_statement: str, goals: list, features: list[Feature], user_stories: list, roadmap: list).

4. System Architect Llama 3.1 Nemotron 70B @ NVIDIA NIM

Generates PostgreSQL syntax and valid Mermaid.js diagrams, declaring scale ceilings. Outputs: ArchitectureSpec (db_schema_sql: str, db_schema_mermaid: str, api_endpoints: list, design_notes: str).

5. Engineering Manager Llama 3.3 70B @ Groq + GitHub API

Creates 4 dependency-ordered sprints using Fibonacci story points, and auto-syncs issues to GitHub in background threads. Outputs: IssuesAndSprintPlan (issues: list[GitHubIssue], sprints: list, definition_of_done: list, risks: list).

6. Marketing Specialist Llama 3.3 70B @ Groq

Applies the PAS (Problem-Agitate-Solve) copy framework, outlines a 5-step email drip campaign, and sets up tier packages. Outputs: MarketingAssets (landing_copy: str, linkedin_post: str, pricing_tiers: list, email_sequence: list).

4. Split-Database Architecture

To balance high-frequency execution checkpoints with permanent data resilience, Blueprint operates a split-database architecture.

SHORT-TERM: SQLite (LangGraph Checkpointer)

LangGraph needs to log the step-by-step state of execution threads (to support pauses/resumes). This is written to checkpoints.db. Since this database is file-based and local, if the hosting server restarts, it is wiped.

LONG-TERM: Supabase PostgreSQL (Permanent Artifacts)

To ensure that final results and history survive server restarts, all validated artifacts and logging statements are pushed permanently to Supabase.

SQLite Fallback: If Supabase credentials are missing or the connection fails, the system automatically falls back to local tables in founder_os.db, guaranteeing service uptime.

5. Secure OAuth Exchange & Integrations

Blueprint does not save user API keys on the server. Instead, it utilizes secure OAuth 2.0 flows to interact directly with the user's accounts.

The GitHub Exchange Sequence

1. The user clicks "Connect GitHub". The frontend redirects to GitHub's authorization page.
2. GitHub redirects back with a temporary ?code=... value.
3. Next.js forwards this code to FastAPI, which exchanges it with GITHUB_CLIENT_SECRET for a secure access_token.
4. The token is stored locally in the browser's localStorage. It is sent as a payload header in execution requests and is never saved in the database.

Notion Block Parsing Complexity

Notion's API does not support standard HTML or Markdown streams. In notion.py, we parse agent artifacts into structured Notion blocks (e.g. bulleted_list_item, quote). Because Notion limits append operations to 100 blocks per request, our backend automatically splits compiled blocks into batches of 80 before making sequential PATCH requests, avoiding API rate limits.

6. Frontend Architecture & React Flow ERD

The frontend is built as a highly responsive Next.js 15 SPA utilizing polling and canvas rendering.

A. Polling and Auto-Advance

The SessionDashboard component polls the backend status every 1.5 seconds. While running, the UI automatically advances active tabs to match the backend progress, keeping the user in sync with the agent stream.

B. React Flow ERD Parser

Instead of rendering plain Mermaid text, a custom Javascript parser in the dashboard extracts SQL entities using regex: 

/(\w+)\s*\{([^}]*)\}/g

Nodes are positioned using the dagre layout engine and rendered as custom table nodes in React Flow, highlighted with primary and foreign key indicators.

7. PDF Generation & Asynchronous Threading

Blueprint exports VC evaluation memos using the xhtml2pdf engine.

A. xhtml2pdf Constraints

Since xhtml2pdf is built on ReportLab, it does not support modern CSS Flexbox or Grid layouts. All layouts (such as SWOT grids and pricing tables) are structured using raw HTML <table> elements. Word wrapping is enforced via a custom function to prevent long strings from breaking margins.

B. Non-Blocking Event Loop

PDF generation is CPU-bound and synchronous. To prevent it from blocking FastAPI's single-threaded event loop, it is delegated to a separate thread pool using: 

await asyncio.to_thread(export_to_pdf, ...)

8. Deployment & Infrastructure

The production architecture is deployed on cloud-native serverless hosting.

  • Frontend: Vercel (Zero-config deployment for Next.js).
  • Backend: Render Web Services (Python environment).
  • Cold Start Mitigation: Render's free tier spins down after 15 minutes of inactivity. To prevent a 30-second delay for judges, UptimeRobot is configured to ping the backend's /docs endpoint every 5 minutes to keep it warm.

Hard Engineering Q&A Checklist

      


      

        
        

          

            Q: How do you handle LLM hallucinations in JSON structures?
            
          

          

            
A: We enforce structure at multiple layers:

            
    
              
  • First, we configure Groq/API parameters with response_format={"type": "json_object"}.
    • 
              
  • Second, we pass the exact Pydantic schema in the system prompt instructions.
    • 
              
  • Third, the backend validates the raw string using schema(**parsed). If keys are missing, Pydantic raises a ValidationError. We catch this in our call_with_retry loop, which automatically retries the LLM request up to 3 times with the validation error appended to the prompt history before failing the stage gracefully.
    • 
            

          

        


        

          

            Q: Why didn't you use Server-Sent Events (SSE) for streaming instead of polling?
            
          

          

            
A: Polling at 1.5-second intervals was a deliberate design choice. Server-Sent Events (SSE) require keeping open connections active on the server. On standard cloud serverless tiers, idle connections are aggressively terminated, leading to connection loss. Since our multi-agent pipeline completes in under 30 seconds, polling results in about 20 requests total. This puts minimal load on Supabase while significantly simplifying the backend state management.

          

        


        

          

            Q: What happens if the backend server restarts in the middle of a pipeline run?
            
          

          

            
A: Since LangGraph checkpointers are stored in a local SQLite file (checkpoints.db) on the server, a restart will terminate the active execution thread. However, any agent stages that completed before the restart are already permanently saved in Supabase. When the user refreshes, the frontend client queries the status. The backend detects that the LangGraph state is missing, queries Supabase to reconstruct the completed artifacts, and loads the session dashboard seamlessly.

          

        


        

          

            Q: Why did you use SQLite for LangGraph checkpoints but Supabase for artifacts?
            
          

          

            
A: LangGraph requires synchronous, low-latency checkpoint logging to save thread state at every single node transition. Saving checkpoints to a remote database like Supabase would add significant network latency to every step. By using local SQLite for the short-term checkpointer, the graph transitions instantly. Once the entire pipeline finishes, the final compiled artifacts are saved to Supabase to guarantee permanent storage.
