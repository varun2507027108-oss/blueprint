Why LangGraph?

              

                Traditional AI agent systems rely on basic sequential execution chains (e.g. LLM 1 output directly flows to LLM 2 input). While simple, this approach fails under production requirements like conditional loops, human interventions, and parallel streams.
              

              

                LangGraph models workflows as stateful Directed Acyclic Graphs (DAGs) and cyclic graphs. It introduces three core primitives that make it superior:
              

              
    
                
  • 
                  
     Shared Graph State
    
                  
    A single type-safe schema (Pydantic model) is passed between all nodes. Agents read exactly what they need and write updates, automatically handled by LangGraph's state reducers.
    
                
    • 
                
  • 
                  
     Human-in-the-Loop Gateways
    
                  
    LangGraph allows thread execution to pause (interrupt) at designated points (e.g., when the Risk Gate is triggered by high risk_scores). The state is checkpointed in SQLite, and execution is resumed using commands.
    
                
    • 
                
  • 
                  
     Native Branching & Merges
    
                  
    In Blueprint, after the Product Manager completes the PRD, execution branches in parallel: the Engineering Stream (Architect and EM) and the Growth Stream (Marketing) execute concurrently. They Fan-In at the join node to synthesize the final reports.
    
                
    • 
              

            

            

              
Maestro-level Performance

              

                By using LangGraph, Blueprint avoids state pollution and allows different nodes to execute on different LLMs that are suited for that task:
              

              

                
Node & LLM Specialisation

                

                  Startup Advisor, Researcher, EM, Marketing: Groq Llama 3.3 70B (ultra-fast inference, high instruction adherence).
                

                

                  Product Manager: Google Gemini 2.5 Flash (massive context window, ideal for cross-referencing research logs).
                

                

                  System Architect: NVIDIA NIM Nemotron 70B (high technical precision, perfect for generating databases and schemas).
                

              

              

                This multi-provider routing represents a major optimization over single-model systems. The system avoids latency bottlenecks by using Groq for rapid-response nodes, while utilizing specialized engines where reasoning depth or context scale is mandatory.
              

            

          

        


        
        

          

            

              
JSON as the Architectural Glue

              

                In a complex pipeline involving three separate API providers (Groq, Google, NVIDIA) and multiple databases, standard raw string outputs create fragile boundaries. Blueprint resolves this by using structured JSON as the universal data contract.
              

              

                JSON plays a critical role in four distinct areas of the system:
              

              
    
                
  • 
                  
     LLM Output Validation
    
                  
    All LLM prompt calls enforce JSON schema formatting. Groq is queried with JSON mode enabled, and the resulting JSON string is parsed and validated against custom Pydantic schemas in models.py, ensuring no malformed outputs break the graph.
    
                
    • 
                
  • 
                  
     Session Persistence
    
                  
    In db.py, when save_artifact is called, the state object is converted to JSON string and stored in the database. When a session is resumed or requested, these JSON columns are deserialized back into Pydantic models.
    
                
    • 
                
  • 
                  
     Client Communication
    
                  
    FastAPI routes serialize internal data structures into clean REST JSON responses. The Next.js frontend uses these models to display state, update loaders, and populate logs.
    
                
    • 
              

            

            

              
Pydantic State Model

              

                Below is the core GraphState definition, showing how Pydantic annotations govern the state and merge properties across parallel node streams.
              

              
class GraphState(BaseModel):
    session_id: str
    startup_name: str
    idea: str
    github_repo: str
    status: str = "running"
    
    # Reducers merge dictionaries and lists when parallel nodes complete
    stages: Annotated[Dict[str, Any], reduce_dict] = Field(default_factory=dict)
    failed_stages: Annotated[List[str], reduce_list] = Field(default_factory=list)

    # Intermediate and final outputs
    startup_advisor: Optional[ValidationResult] = None
    market_research: Optional[MarketResearchReport] = None
    product_manager: Optional[PRD] = None
    architect: Optional[ArchitectureSpec] = None
    engineering_manager: Optional[IssuesAndSprintPlan] = None
    marketing: Optional[MarketingAssets] = None

    gate_decision: Optional[str] = None
    revised_idea: Optional[str] = None
    github_token: Optional[str] = None

            

          

        


        
        

          

            

              
PDF Compiling via xhtml2pdf

              

                When the graph reaches completion, a final report is compiled. The system uses xhtml2pdf (also known as pisa) to convert HTML and CSS files directly to a printable PDF.
              

              

                Due to internal engine limitations in xhtml2pdf (which is built on top of ReportLab), standard modern CSS layouts will fail. The compiler enforces strict guidelines:
              

              

                
 Engine Hard Limits

                

                  No Flexbox / Grid: Flex and grid layouts are completely unsupported. All multi-column outputs (e.g. side-by-side callouts) must be written as legacy HTML <table> structures.
                

                

                  No White-Space wrapping: CSS white-space: pre-wrap fails to wrap text. Code blocks and long comments must be pre-processed and force-wrapped via string utilities.
                

                

                  Page Margins & Breaks: Headers, footers, and page numbers must be defined using custom CSS page margins (@page declarations and counters).
                

              

            

            

              
HTML-to-PDF Pipeline

              

                The code inside pdf_export.py builds a single document:
              

              
    
                
  • 
                  
    1. Text Sanitisation
    
                  
    Special characters are HTML-escaped. Newline characters (\n) are replaced with <br/>, and long strings are divided by line wraps to avoid clipping at the page edge.
    
                
    • 
                
  • 
                  
    2. Micro-component Tables
    
                  
    Badges (e.g. Must-Have, High Priority) are rendered using table cells with solid background colors and inline styles. Column splits are restricted to width tables (e.g., width: 50%).
    
                
    • 
                
  • 
                  
    3. Pisa Rendering
    
                  
    The completed HTML string is loaded, and pisa.CreatePDF() writes the binary output. If any rendering error occurs, the partial file is safely deleted before throwing an exception.
    
                
    • 
              

            

          

        


        
        

          

            

              
GitHub Issues Sync

              

                When a founder supplies a github_repo and a valid token, the Engineering Manager node triggers a background sync to create the project's task backlog automatically.
              

              

                To keep the pipeline fast and prevent timeouts, the issues are synchronized asynchronously:
              

              

                
 Non-blocking Background Tasks

                

                  Once the sprint plan is generated, the EM node spawns a background asyncio task (create_github_issues_bulk) to call the GitHub API.
                

                

                  This avoids blocking the FastAPI response thread. If the sync succeeds or fails, a log is written to the decision ledger, visible to the frontend in the logs drawer.
                

              

            

            

              
Notion Database Exporter

              

                The Notion integration allows founders to save their startup specs into a unified database. The process involves structural block translation:
              

              
    
                
  • 
                  
    1. Page Initialization
    
                  
    The system contacts api.notion.com using OAuth credentials to create a new page under the specified Notion database ID.
    
                
    • 
                
  • 
                  
    2. Block Construction
    
                  
    In notion.py, state artifacts are translated into Notion block objects. Headers, lists, and quotes are converted. Large SQL DDL schema files are placed inside an interactive toggle block with code formatting.
    
                
    • 
                
  • 
                  
    3. Chunks Appending
    
                  
    Notion limits page appends to 100 blocks per request. The system splits the assembled blocks into chunks of 80 and makes sequential PATCH requests to avoid hitting API rate limits.
